# 🛡️ RealGuard — Real-Time Financial Fraud Detection System

> A production-grade, end-to-end ML pipeline that ingests live financial transactions, engineers real-time features, runs a stacked ensemble model, serves sub-200ms predictions with SHAP explanations, and visualises latency and throughput on a live Grafana dashboard — all running locally via Docker Compose.

---

## 📌 Project Summary

**RealGuard** is not a Kaggle notebook. It is a complete ML *system* — every component you would find in a real fintech company's fraud stack, implemented from scratch, containerised, and demonstrable in a single command.

| Property | Detail |
|---|---|
| Dataset | PaySim Synthetic Financial Dataset (6.3M transactions) |
| Primary metric | Recall @ 95% Precision (fraud catch rate) |
| Inference latency | Target < 200ms end-to-end |
| Serving | REST API with fraud score + SHAP reason codes |
| Monitoring | Live Grafana dashboard (latency, throughput, fraud rate) |
| Deployment | Docker Compose — one command, full system up |

## 🧰 Tool Stack — Full Explanation

Every tool is listed below with what it does, why it was chosen, and where it lives in the codebase. Read this before asking Antigravity to build anything.

---

### 1. Dataset — PaySim Synthetic Financial Dataset

**What it is:** A synthetic dataset of 6.3 million mobile money transactions generated using real transaction logs from a mobile money service in Africa. Contains transfers, payments, cash-ins, and cash-outs with a ground-truth `isFraud` label.

**Why PaySim:** It is large enough (6.3M rows) to make data engineering decisions meaningful, has realistic fraud patterns (only ~0.13% fraud — severe class imbalance), and is freely available on Kaggle with no licensing restrictions.

**Download:** `https://www.kaggle.com/datasets/ealaxi/paysim1`
**Save to:** `data/raw/PS_20174392719_1491204439457_log.csv`

**Key columns:**
| Column | Description |
|---|---|
| `step` | Time unit (1 step = 1 hour, total 744 steps = 30 days) |
| `type` | Transaction type: TRANSFER, CASH_OUT, PAYMENT, etc. |
| `amount` | Transaction amount |
| `nameOrig` | Origin account ID |
| `oldbalanceOrg` | Origin balance before transaction |
| `newbalanceOrig` | Origin balance after transaction |
| `nameDest` | Destination account ID |
| `isFraud` | Ground truth label (1 = fraud, 0 = legitimate) |

---

### 2. Apache Kafka — Event Streaming Backbone

**What it is:** A distributed event streaming platform. Think of it as a high-throughput message bus. Producers write messages (transactions) to named topics. Consumers read from those topics in real time.

**Why Kafka:** Every real fintech system uses Kafka (or something equivalent) as the backbone for transaction data. It decouples the data source from the processing layer, handles backpressure, and lets multiple consumers read the same stream independently.

**How it's used here:**
- `ingestion/producer.py` reads the PaySim CSV row by row and publishes each transaction as a JSON message to the Kafka topic `raw-transactions`, with a configurable delay to simulate real-time (e.g. 100 transactions/second)
- The Faust streaming app consumes from this same topic

**Key config:**
```python
KAFKA_BROKER = "kafka:9092"          # Inside Docker network
KAFKA_TOPIC = "raw-transactions"
PRODUCE_RATE = 100                   # transactions per second
```

**Docker service name:** `kafka`
**Port:** `9092`
**Managed by:** Confluent's `cp-kafka` Docker image (includes Zookeeper)

---

### 3. Faust — Python Stream Processing

**What it is:** A Python-native stream processing library built on top of Kafka. It lets you write async Python functions that react to every Kafka message, maintain stateful in-memory tables, and compute windowed aggregations — all without touching Java or Scala.

**Why Faust over Spark Streaming / PyFlink:** Spark Streaming works in micro-batches (1–5 second windows), which means features are always slightly stale — unacceptable for real-time fraud. PyFlink requires Java 11, complex JAR configuration, and takes days to set up. Faust is pure Python, installs via pip, and delivers true per-event processing within seconds of installation.

**How it's used here:**
`streaming/faust_app.py` consumes `raw-transactions` and computes the following features per `nameOrig` (card/account ID) in sliding windows:

| Feature | Window | Description |
|---|---|---|
| `txn_count_5m` | 5 minutes | Number of transactions in last 5 min |
| `txn_count_1h` | 1 hour | Number of transactions in last 1 hour |
| `avg_amount_1h` | 1 hour | Average transaction amount last 1 hour |
| `max_amount_1h` | 1 hour | Maximum transaction amount last 1 hour |
| `unique_dest_1h` | 1 hour | Unique destination accounts last 1 hour |
| `balance_drop_pct` | Per event | % drop in origin balance this transaction |
| `txn_count_24h` | 24 hours | Total transactions in last 24 hours |

These computed features are immediately written to Redis after each event.

**Install:** `pip install faust-streaming` (maintained fork of original Faust)

---

### 4. Redis — Real-Time Feature Store

**What it is:** An in-memory key-value store. Reads and writes happen in under 1 millisecond. Used here as the online feature store — the bridge between the streaming layer (Faust computes features) and the serving layer (FastAPI reads features at inference time).

**Why Redis:** At inference time, FastAPI needs the pre-computed windowed features for an incoming transaction's account ID. Querying a database would take 5–50ms, blowing the latency budget. Redis serves the same data in <1ms.

**Key pattern used:**
```
SET feat:{account_id}  →  JSON blob of all features for that account
GET feat:{account_id}  →  retrieved at inference time in FastAPI
TTL: 86400 seconds (24 hours — features expire after 1 day of inactivity)
```

**Docker service name:** `redis`
**Port:** `6379`
**Python client:** `redis-py`

---

### 5. XGBoost — Primary Gradient Boosting Classifier

**What it is:** Extreme Gradient Boosting. An ensemble of decision trees trained sequentially, where each tree corrects the errors of the previous one. The gold standard for tabular classification tasks.

**Why XGBoost:** Handles class imbalance well via `scale_pos_weight`, trains fast, produces well-calibrated probabilities, and is natively supported by SHAP for explainability. On the IEEE-CIS and PaySim benchmarks, XGBoost consistently ranks in the top-3 approaches.

**How it's used here:**
- Trained in `models/train_xgboost.py`
- Features: raw transaction fields + windowed features from Redis (at training time, pre-computed via batch)
- Class imbalance handled via `scale_pos_weight = (n_legitimate / n_fraud)`
- Hyperparameters tuned via Optuna (50 trials)
- Output: fraud probability score between 0 and 1
- Logged to MLflow with all hyperparameters and metrics

**Key hyperparameters tuned:**
`n_estimators`, `max_depth`, `learning_rate`, `subsample`, `colsample_bytree`, `min_child_weight`

---

### 6. LightGBM — Secondary Gradient Boosting Classifier

**What it is:** Light Gradient Boosting Machine by Microsoft. Similar to XGBoost but uses a leaf-wise tree growth strategy instead of level-wise, making it faster to train on large datasets and often achieving slightly better accuracy.

**Why LightGBM alongside XGBoost:** The two models make different errors — XGBoost tends to perform better on smaller feature sets, LightGBM on larger ones. Running both and stacking their outputs captures the best of both. Training LightGBM on 6.3M rows takes ~2–3 minutes — practical for this project.

**How it's used here:**
- Trained in `models/train_lightgbm.py`
- Same feature set as XGBoost
- Class imbalance handled via `is_unbalance=True`
- Hyperparameters tuned via Optuna (50 trials)
- Output: fraud probability score, passed to meta-learner

---

### 7. PyTorch LSTM — Sequential Pattern Detector

**What it is:** Long Short-Term Memory network — a type of recurrent neural network designed to learn patterns in sequences. Instead of treating each transaction independently, LSTM looks at the last N transactions for an account and detects patterns in the sequence.

**Why LSTM:** Fraudsters follow behavioural sequences — small test transaction → large transfer → cash out. XGBoost and LightGBM treat each transaction independently and cannot detect these patterns. LSTM sees the sequence and catches this behaviour.

**How it's used here:**
- Input: last 10 transactions for `nameOrig`, each encoded as a feature vector
- Architecture: 2-layer LSTM → Dropout(0.3) → Linear → Sigmoid
- Trained in `models/train_lstm.py` using PyTorch
- For accounts with fewer than 10 transactions, sequences are zero-padded
- Output: fraud probability score, passed to meta-learner

**Sequence features per timestep:**
`amount`, `type_encoded`, `hour_of_day`, `balance_drop_pct`, `avg_amount_1h`

---

### 8. Stacking Ensemble — Logistic Regression Meta-Learner

**What it is:** A technique where multiple base models (XGBoost, LightGBM, LSTM) each produce a probability score, and a second-level model (the meta-learner) takes all three scores as input and produces the final fraud probability.

**Why stacking:** Each base model captures different patterns. Stacking lets the meta-learner learn *when to trust which model*. On imbalanced fraud datasets, stacking consistently outperforms any single model by 3–8% on recall at fixed precision.

**How it's used here:**
- Meta-learner: Logistic Regression (scikit-learn)
- Inputs: `[xgb_score, lgbm_score, lstm_score]`
- Trained in `models/train_meta.py` using out-of-fold predictions to prevent data leakage
- Final output: single fraud probability between 0 and 1
- Decision threshold tuned to maximise recall at 95% precision

**Training order:**
1. Train XGBoost, LightGBM, LSTM on 80% train set
2. Generate out-of-fold predictions on 20% validation set
3. Train Logistic Regression meta-learner on those out-of-fold predictions
4. All four models registered in MLflow

---

### 9. Imbalanced-learn — Class Imbalance Handling

**What it is:** A Python library extending scikit-learn with algorithms for handling imbalanced datasets.

**Why needed:** PaySim has ~0.13% fraud. If you train without handling this, the model learns to predict "not fraud" for everything and achieves 99.87% accuracy while being completely useless.

**How it's used here:**
- SMOTE (Synthetic Minority Oversampling Technique): generates synthetic fraud samples in feature space to balance the training set
- Used in training pipelines for XGBoost and LightGBM
- LSTM uses class-weighted loss instead (`pos_weight` parameter in BCEWithLogitsLoss)

---

### 10. Optuna — Hyperparameter Optimisation

**What it is:** An automatic hyperparameter optimisation framework. Defines a search space, runs trials, and uses Bayesian optimisation (TPE sampler) to find the best hyperparameters efficiently.

**Why Optuna:** Grid search over 6 hyperparameters with 5 values each = 15,625 combinations. Optuna finds near-optimal parameters in 50 trials by intelligently sampling the search space.

**How it's used here:**
- Separate Optuna study for XGBoost and LightGBM
- 50 trials each, optimising for `recall` at 95% precision on validation set
- Best parameters automatically logged to MLflow
- Results stored in `optuna_studies/` as SQLite databases (so you can resume)

---

### 11. SHAP — Explainability

**What it is:** SHapley Additive exPlanations. A game-theory-based method for explaining any ML model's predictions. For each prediction, SHAP assigns a contribution score to every feature — positive means "pushed toward fraud", negative means "pushed toward legitimate".

**Why SHAP:** Black-box fraud scores are useless in practice. Banks and regulators require reason codes — "why was this transaction flagged?" SHAP provides this. It also helps you debug the model during development (if the model is using suspicious features, SHAP shows you).

**How it's used here:**
- `explainability/shap_explainer.py` uses `shap.TreeExplainer` for XGBoost and LightGBM (fast, exact SHAP values)
- At inference time, FastAPI returns the top-3 SHAP reason codes in the API response
- `notebooks/04_shap_analysis.ipynb` contains global SHAP summary plots, dependence plots, and waterfall charts for the README visuals

**Example API response with SHAP:**
```json
{
  "transaction_id": "T-00291847",
  "fraud_probability": 0.94,
  "decision": "BLOCK",
  "latency_ms": 143,
  "reason_codes": [
    {"feature": "balance_drop_pct", "impact": +0.38, "value": 0.99},
    {"feature": "unique_dest_1h", "impact": +0.21, "value": 7},
    {"feature": "txn_count_5m", "impact": +0.18, "value": 12}
  ]
}
```

### 12. MLflow — Experiment Tracking + Model Registry

**What it is:** An open-source platform for managing the ML lifecycle. Tracks experiments (hyperparameters, metrics, artifacts), stores trained models, and maintains a registry of production-ready model versions.

**Why MLflow:** Without experiment tracking, ML development is chaos — you can't remember which parameters gave the best recall, which model version is in production, or what the baseline was. MLflow makes your development process auditable and professional.

**How it's used here:**
- Every training run logs: all hyperparameters, train/val metrics (precision, recall, AUC, F1), confusion matrix, feature importance plot
- Four models registered: `fraud-xgboost`, `fraud-lightgbm`, `fraud-lstm`, `fraud-stacking-ensemble`
- FastAPI's `predictor.py` loads the `fraud-stacking-ensemble` model tagged `Production` from the registry at startup
- MLflow UI accessible at `http://localhost:5000`

**Docker service name:** `mlflow`
**Port:** `5000`

### 13. FastAPI — Inference API

**What it is:** A modern, high-performance Python web framework for building APIs. Based on Python type hints — request and response schemas are defined as Pydantic models, giving automatic validation and OpenAPI docs.

**Why FastAPI:** Already in Arjun's stack (DocMind uses FastAPI). Async support means it can handle concurrent inference requests without blocking. Auto-generates Swagger docs at `/docs`.

**How it's used here:**

**Endpoint: `POST /predict`**
```
Request  → transaction fields (amount, type, account IDs, balances)
Processing →
  1. Look up windowed features from Redis by account ID
  2. Merge with transaction fields to build feature vector
  3. Run XGBoost, LightGBM, LSTM in parallel
  4. Pass scores to meta-learner → final probability
  5. Run SHAP on XGBoost output → top-3 reason codes
  6. Record latency, increment Prometheus counters
Response → fraud_probability, decision, latency_ms, reason_codes
```

**Endpoint: `GET /health`** — liveness check
**Endpoint: `GET /metrics`** — Prometheus scrape endpoint

**Docker service name:** `api`
**Port:** `8000`
**Swagger UI:** `http://localhost:8000/docs`

### 16. Evidently AI — Data Drift Detection (Manual)

**What it is:** An open-source ML monitoring library that generates HTML reports comparing two datasets — typically your training data vs recent production data — and detects statistical drift.

**Why Evidently (manual mode):** Full automated retraining pipelines (Airflow) are production-level complexity beyond this project's scope. Instead, `monitoring/drift_check.py` is a standalone script you run weekly. It compares the last 10,000 production transactions against the training distribution, generates an HTML drift report, and prints a warning if PSI (Population Stability Index) > 0.2 on any key feature. This demonstrates MLOps awareness without requiring full Airflow setup.

**Run drift check:**
```bash
python monitoring/drift_check.py --reference data/processed/train.parquet --current data/processed/recent_production.parquet
# Outputs: monitoring/reports/drift_report_YYYYMMDD.html
```

**Manual retraining trigger:**
```bash
python models/retrain.py --data data/raw/PS_20174392719_1491204439457_log.csv
# Retrains all models, logs to MLflow, promotes best ensemble to Production
```

### Phase 4 — Feature Engineering (Batch, for Training)

**Goal:** Generate the full feature matrix for training all three models.

**Notebook:** `notebooks/02_feature_engineering.ipynb`

**Features to engineer (beyond windowed):**
- `balance_drop_pct` = `(oldbalanceOrg - newbalanceOrig) / (oldbalanceOrg + 1)`
- `amount_to_balance_ratio` = `amount / (oldbalanceOrg + 1)`
- `is_large_transfer` = 1 if `type == TRANSFER` and `amount > 200000`
- `dest_balance_increased` = 1 if `newbalanceDest > oldbalanceDest`
- `hour_of_day` = `step % 24`
- `day_of_month` = `step // 24`
- One-hot encode `type` column

**Save output:** `data/processed/features.parquet`

**Deliverable:** `features.parquet` with all engineered features, shape logged to notebook, no nulls.

**Goal:** Live Grafana dashboard showing real-time latency and throughput.

**Build order:**
1. `monitoring/prometheus.yml` — scrape config pointing to `api:8000/metrics`
2. Start Prometheus and Grafana: `docker compose up prometheus grafana -d`
3. Import `monitoring/grafana/dashboard.json` into Grafana
4. Run the Kafka producer at 100 TPS: `python ingestion/producer.py --rate 100`
5. Watch dashboard update in real time

**Deliverable:** Live dashboard with at minimum — requests/sec, P95 latency, fraud rate gauge — all updating in real time.

### Phase 9 — Full System Integration Test

**Goal:** Everything running together in Docker Compose with one command.

**Steps:**
1. `docker compose up --build` — starts all 8 services
2. Wait ~60 seconds for all services to initialise
3. `python ingestion/producer.py --rate 100` — start transaction stream
4. Open `http://localhost:8000/docs` — confirm API live
5. Open `http://localhost:5000` — confirm MLflow showing experiments
6. Open `http://localhost:3000` — confirm Grafana dashboard live with data
7. Send 10 manual test predictions via Swagger UI
8. Verify SHAP reason codes in every response

**Deliverable:** Screenshot of all four UIs open simultaneously for the README demo section.
